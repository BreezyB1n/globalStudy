# GlobalStudy AI 子任务 1 技术方案

## 1. 子任务定位

### 1.1 子任务名称

项目骨架与基础设施方案

### 1.2 子任务目标

在没有任何业务代码的前提下，先建立一个可运行、可扩展、可联调的本地单体应用骨架，为后续数据采集、知识库构建、问答服务和前端展示提供统一运行底座。

### 1.3 完成标准

- 后端服务可以本地启动
- 前端静态页面可以通过后端托管访问
- 配置项通过环境变量统一管理
- 日志、异常处理、健康检查具备基础能力
- 为后续脚本、数据库、向量库预留固定目录结构

## 2. 负责范围

本子任务只处理基础设施，不处理业务功能。

纳入范围：

- FastAPI 应用入口
- 配置管理
- 日志体系
- 统一响应与异常处理
- 静态资源托管
- 基础目录初始化
- 本地开发启动方式

不纳入范围：

- 官网抓取逻辑
- SQLite 建表与业务写入
- ChromaDB 写入
- LangGraph 工作流
- 前端聊天交互

## 3. 推荐目录结构

```text
fyp/
  app/
    main.py
    core/
      config.py
      logging.py
      exceptions.py
    api/
      health.py
    schemas/
      common.py
  frontend/
    index.html
    styles.css
    app.js
  scripts/
  data/
    raw/
    processed/
    sqlite/
    chroma/
  logs/
  .env.example
  requirements.txt
  README.md
```

设计理由：

- `app/` 只放运行时代码
- `frontend/` 与后端逻辑分离，但最终由 FastAPI 托管
- `data/` 统一承载抓取产物、处理结果和本地数据库
- `scripts/` 专门放离线处理脚本，避免和在线服务混在一起

## 4. 核心设计

### 4.1 应用启动方式

推荐采用单入口模式：

- `app/main.py` 负责创建 FastAPI 应用
- 挂载 `/api` 路由
- 挂载前端静态资源目录
- 暴露 `/health` 健康检查接口

推荐本地启动命令：

```bash
uvicorn app.main:app --reload
```

### 4.2 配置管理

推荐使用 `pydantic-settings` 或等价方案集中管理配置。

至少包含以下配置项：

- `APP_NAME`
- `APP_ENV`
- `APP_HOST`
- `APP_PORT`
- `BAILIAN_API_KEY`
- `BAILIAN_BASE_URL`
- `FIRECRAWL_API_KEY`
- `SQLITE_PATH`
- `CHROMA_PERSIST_DIR`
- `RAW_DATA_DIR`
- `PROCESSED_DATA_DIR`
- `LOG_LEVEL`

设计要求：

- 所有配置项必须有明确来源
- 开发环境允许读取 `.env`
- 代码中禁止硬编码 API Key
- 路径类配置统一转成绝对路径或项目根目录相对路径

### 4.3 日志设计

日志至少分为三类：

- 应用启动日志
- 接口调用日志
- 脚本执行日志

建议日志字段：

- `timestamp`
- `level`
- `module`
- `message`
- `request_id`
- `path`
- `duration_ms`

要求：

- 前后端异常必须在日志中可定位
- 脚本执行失败时能定位具体学校、项目或 URL
- 后续问答服务可以复用同一套日志格式

### 4.4 异常处理

需要建立统一异常出口，避免接口直接抛出未处理堆栈。

建议区分：

- 参数错误
- 第三方 API 调用失败
- 本地文件不存在
- 数据库读写失败
- 知识库尚未建立

响应结构建议：

```json
{
  "code": "KNOWLEDGE_BASE_NOT_READY",
  "message": "知识库尚未构建，请先执行重建脚本",
  "request_id": "..."
}
```

### 4.5 静态资源托管

第一版前端很轻，建议直接由 FastAPI 托管：

- `/` 返回 `frontend/index.html`
- `/assets/*` 或等价目录承载 CSS、JS

这样可以减少本地演示时的启动步骤，避免同时开两个服务。

## 5. 后续模块预留接口

### 5.1 API 路由预留

建议预留以下命名空间：

- `/api/health`
- `/api/chat`
- `/api/admin/rebuild` 或后续改为脚本不暴露接口

其中本阶段只需要实现健康检查，其他接口保留目录结构即可。

### 5.2 数据目录预留

后续模块会依赖以下路径：

- `data/raw/` 存放抓取原始页面
- `data/processed/` 存放结构化抽取结果和中间文件
- `data/sqlite/` 存放 `app.db`
- `data/chroma/` 存放 ChromaDB 持久化目录

### 5.3 脚本执行预留

后续脚本统一要求：

- 能从项目根目录启动
- 能读取同一套配置项
- 能复用同一套日志模块

## 6. 技术约束

### 6.1 开发约束

- 只做单体本地方案，不引入 Docker 作为第一版必要条件
- 不引入 Redis、PostgreSQL、消息队列等额外基础设施
- 保持目录清晰，避免为未来扩展过度设计

### 6.2 可扩展约束

虽然第一版轻量，但要保证以下扩展空间：

- 后续可以增加更多 API 路由
- 后续可以替换前端为更复杂方案，但不影响后端接口
- 后续可以替换模型提供方，但不影响配置结构

## 7. 风险与处理

### 7.1 风险一：应用目录过早混乱

问题：

如果一开始就把脚本、API、数据模型、前端文件混放，后期接入 LangGraph 和知识库时会快速失控。

处理：

- 强制按职责拆目录
- 在线服务和离线脚本分离
- 配置、日志、异常处理先统一

### 7.2 风险二：环境变量分散

问题：

抓取脚本、问答服务、建库脚本如果各自读取不同配置，排错成本会很高。

处理：

- 所有运行入口统一读取 `config.py`
- `.env.example` 明确列出必填项

## 8. 验收标准

- 启动命令执行后，浏览器可访问首页
- `/api/health` 返回服务正常
- 缺失关键 API Key 时，启动阶段能给出明确错误提示
- 访问不存在的 API 时能返回统一错误结构
- `data/`、`logs/` 等基础目录具备明确约定

## 9. 本子任务交付结果

完成本子任务后，项目应具备“应用能跑起来”的最低工程基础。后续所有功能都应建立在这套骨架之上，而不是绕开它各自实现一套运行方式。
