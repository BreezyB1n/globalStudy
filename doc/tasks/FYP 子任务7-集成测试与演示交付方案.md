# GlobalStudy AI 子任务 7 技术方案

## 1. 子任务定位

### 1.1 子任务名称

集成测试与演示交付方案

### 1.2 子任务目标

在各模块具备基础功能后，建立全量重建脚本、联调流程、测试清单、验收标准和演示方案，确保 Demo 在毕业设计场景下可稳定运行、可稳定展示、可清晰说明。

### 1.3 完成标准

- 可以从空数据状态完整重建系统
- 关键链路具备最小测试覆盖
- 演示流程清晰可复现
- 风险场景有备用方案

## 2. 负责范围

纳入范围：

- 全量与局部重建入口
- 基础测试设计
- 联调顺序
- Demo 数据准备
- 答辩演示脚本

不纳入范围：

- 单个业务模块内部实现
- 额外的生产运维能力

## 3. 集成目标

系统最终需要跑通以下完整链路：

```text
维护入口清单
-> 官网抓取
-> 原始文件落盘
-> 结构化抽取入 SQLite
-> 文本切分入 ChromaDB
-> 启动 FastAPI
-> 前端提问
-> LangGraph 检索与回答
-> 展示证据与原始链接
```

任何一个环节无法复现，这个 Demo 在答辩时都会显得不完整，因此本任务的核心是把整个链路固定成可重复执行的标准流程。

## 4. 重建脚本设计

### 4.1 全量重建

建议脚本：

```text
scripts/rebuild_all.py
```

建议执行顺序：

1. 校验配置项
2. 读取入口清单
3. 调用抓取脚本
4. 调用结构化抽取脚本
5. 调用向量建库脚本
6. 输出汇总报告

### 4.2 局部重建

建议支持：

- 指定学校重建
- 指定项目重建
- 指定页面类型重抓后再重建

价值：

- 局部修正成本低
- 答辩前修复某个项目数据更方便

## 5. 测试策略

### 5.1 单元测试重点

建议优先覆盖：

- 配置加载
- URL 清单解析
- 结构化字段标准化
- chunk 切分逻辑
- 问题类型路由
- 引用结构格式化

### 5.2 集成测试重点

建议至少覆盖以下链路：

- 选定一个项目，从原始文件到 SQLite 写入成功
- 同一个项目，从原始文件到 ChromaDB 写入成功
- `/api/chat` 能返回答案与引用
- 前端页面能正确渲染一轮问答结果

### 5.3 人工验收测试

建议准备一组固定问题：

- “介绍一下这个项目”
- “学费是多少”
- “雅思要求多少”
- “申请截止时间是什么时候”
- “和另一个项目相比有什么不同”

每次演示前按这组问题做 smoke test。

## 6. 数据验收标准

### 6.1 项目覆盖度

建议至少选出：

- 10 到 15 所代表性学校
- 每所学校 1 到 2 个代表性硕士项目

不建议盲目追求项目数量，而是优先保证样本质量。

### 6.2 字段完整度

重点检查：

- `program_name`
- `overview`
- `language_requirement`
- `tuition`
- `application_deadline`

这些字段直接决定问答体验。

### 6.3 引用完整度

每条回答都应尽量具备：

- 页面标题
- 原始链接
- 证据片段

如果回答没有任何引用，演示效果会明显下降。

## 7. 联调顺序建议

建议固定按以下顺序联调：

1. 骨架和健康检查
2. 抓取脚本
3. 结构化抽取
4. 向量建库
5. `/api/chat`
6. 前端页面
7. 全量重建脚本

这样可以快速定位问题属于哪一层，而不是在全链路里盲查。

## 8. 毕业设计演示方案

### 8.1 演示目标

答辩演示需要让老师快速看到三个核心点：

- 数据来自学校官网
- 系统能准确回答项目问题
- 回答具备来源证据

### 8.2 推荐演示顺序

1. 先展示系统首页和项目定位
2. 询问一个基础介绍问题
3. 继续追问语言要求或学费
4. 打开引用链接说明来源
5. 再展示一个比较问题

### 8.3 演示样本准备

建议预选 3 到 5 个最稳定项目作为答辩主样本，要求：

- 官网结构清晰
- 字段较完整
- 检索结果质量好

其余学校作为补充样本，不必在答辩时全部展示。

## 9. 风险与处理

### 9.1 风险一：临场网络或 API 不稳定

处理：

- 答辩前提前完成抓取和建库
- 运行时只保留问答调用必要的在线依赖
- 必要时准备已构建好的本地数据目录

### 9.2 风险二：个别问题回答不稳

处理：

- 预先筛选最佳演示样本
- 对高频演示问题做回归测试
- 保证引用能够解释答案来源

### 9.3 风险三：评委追问系统如何保证真实性

处理：

- 直接展示引用区和原始链接
- 说明结构化字段与证据片段的绑定关系
- 说明系统只采集学校官网数据

## 10. 验收标准

- 一条命令或一组固定命令可以从空目录完成重建
- 核心问答链路通过 smoke test
- 演示样本问题回答稳定
- 关键错误场景有明确提示和备用方案
- 项目文档足以支撑实现、联调和答辩说明

## 11. 本子任务交付结果

完成本子任务后，项目将从“几个能跑的模块”变成“一个可演示、可验证、可交付的毕业设计系统”。这一层决定的是最终呈现质量，而不是单点技术实现。
