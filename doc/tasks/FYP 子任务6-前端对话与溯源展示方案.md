# GlobalStudy AI 子任务 6 技术方案

## 1. 子任务定位

### 1.1 子任务名称

前端对话与溯源展示方案

### 1.2 子任务目标

基于原生 HTML、CSS、JavaScript 构建一个轻量单页前端，使用户能够发起多轮提问、查看回答、查看证据片段和原始链接，并在错误或无结果场景下获得清晰反馈。

### 1.3 完成标准

- 页面可本地访问
- 支持连续提问和追问
- 回答区域和引用区域清晰分离
- 加载、失败、无结果状态明确
- 页面结构能稳定支撑毕业设计演示

## 2. 负责范围

纳入范围：

- 单页布局
- 消息列表渲染
- 输入框与发送逻辑
- 会话上下文维护
- 引用卡片展示
- 错误提示和加载状态

不纳入范围：

- 后端问答逻辑
- 用户账号系统
- 历史记录持久化

## 3. 页面结构设计

### 3.1 页面组成

建议页面只保留四个核心区域：

- 顶部标题区
- 中部对话区
- 底部输入区
- 回答下方引用区

### 3.2 推荐布局

```text
--------------------------------
 标题 / 项目名称 / 简介
--------------------------------
 对话消息列表
   用户消息
   助手回答
   证据卡片
--------------------------------
 输入框 + 发送按钮
--------------------------------
```

设计原则：

- 结构简洁，避免答辩时界面信息噪音过大
- 强调“回答”和“来源”这两个最重要的结果

## 4. 前端状态设计

### 4.1 会话状态

前端需维护：

- 当前输入内容
- 消息列表
- 是否正在加载
- 最近若干轮消息
- 当前错误状态

### 4.2 消息结构建议

```json
{
  "role": "assistant",
  "content": "该项目的语言要求为……",
  "citations": []
}
```

说明：

- `assistant` 消息可以挂载 `citations`
- 用户消息只需要 `role` 和 `content`

### 4.3 上下文传递策略

每次调用 `/api/chat` 时：

- 发送最近若干轮消息
- 不做长期本地存档
- 页面刷新后会话可清空

这与整体范围保持一致，避免引入历史管理功能。

## 5. 交互流程设计

### 5.1 发送消息流程

1. 用户输入问题
2. 前端校验输入不为空
3. 将用户消息加入消息列表
4. 显示加载中状态
5. 调用 `/api/chat`
6. 渲染回答和引用
7. 清空输入框并解除加载状态

### 5.2 追问流程

用户继续输入时，前端不需要理解上下文，只负责继续携带最近消息发送给后端。

### 5.3 无结果流程

如果后端返回“未找到明确答案”：

- 仍然显示回答文本
- 如果有引用，继续展示引用
- 如果没有引用，显示“当前问题暂无可展示证据”

## 6. 引用展示设计

### 6.1 引用卡片字段

每条引用卡片建议展示：

- 学校名
- 项目名
- 页面标题
- 证据片段
- 原始链接按钮

### 6.2 展示原则

- 证据片段必须短而清晰
- 链接可以新开标签页查看
- 同一回答下的多条引用按相关性排序

### 6.3 用户感知目标

引用区的目的不是展示技术细节，而是让用户一眼理解：

- 这条回答不是凭空生成的
- 我可以点回官网验证

## 7. 错误与状态设计

### 7.1 加载中

发送请求后需要：

- 禁用发送按钮
- 显示“正在检索官网信息”或等价提示

### 7.2 接口错误

如果接口异常：

- 在对话区显示友好错误消息
- 不要让页面卡死
- 用户可以立即重新提问

### 7.3 知识库未准备好

若后端返回知识库未建立：

- 应明确提示“请先执行数据构建脚本”
- 不要展示模糊的技术堆栈错误

## 8. 文件组织建议

```text
frontend/
  index.html
  styles.css
  app.js
  api.js
  render.js
```

说明：

- 第一版仍然可以只用少量文件
- 但建议至少把 API 调用和渲染逻辑拆开，避免一个 JS 文件过快失控

## 9. 风格与演示要求

### 9.1 视觉方向

前端不需要追求复杂设计，但要满足：

- 排版整洁
- 信息层级清晰
- 适合桌面端演示
- 在移动端也不至于严重错位

### 9.2 文案方向

界面文案建议围绕以下主题：

- “留学项目智能咨询”
- “基于学校官网信息”
- “回答附带来源证据”

这会直接强化答辩时的产品定位。

## 10. 风险与处理

### 10.1 风险一：页面过于简陋，缺少展示感

处理：

- 不增加复杂功能
- 通过清晰排版和引用卡片强化产品完整度

### 10.2 风险二：引用区过长影响阅读

处理：

- 控制每次返回的引用数量
- 每条引用只展示必要字段

### 10.3 风险三：前端保存过多上下文导致请求膨胀

处理：

- 只回传最近若干轮消息
- 长会话可以在前端主动截断

## 11. 验收标准

- 页面可以通过后端静态托管访问
- 用户可连续进行多轮提问
- 每次回答都能正确渲染引用卡片
- 加载、失败、无结果三类状态都能看见明确反馈
- 页面在桌面端演示时布局稳定

## 12. 本子任务交付结果

完成本子任务后，系统将具备完整的用户可见界面。用户不需要理解后端如何工作，只需要看到“提问、回答、查看来源”的闭环即可。
